1 · Access

Sign in, second factor, enrolment

Admin auth is deliberately separate from reader auth. Readers sign in with Google or Apple through Supabase; an admin signs in with an email, a password, and a six-digit code from an authenticator app. Nothing about being a reader grants any admin power, and the admin table is not the profiles table — so a compromised Google account cannot reach moderation.

Notes

What these mockups commit us to

The security model

Admin identity is completely separate from reader identity. Readers are Supabase Auth rows with Google/Apple sign-in; admins are rows in a new admin_users table with an Argon2id password hash and a TOTP secret. There is no path from "I am signed into the app" to "I can moderate", and no admin bootstrap endpoint — the first super admin (at.shamshi@gmail.com) is seeded by a one-off script, everyone after that arrives by invitation.

  • Password — Argon2id, minimum 8 characters (owner decision — TOTP carries the weight).
  • TOTP — RFC-6238, 30-second step, ±1 drift, replay-refused. Enrolment is forced on first sign-in; an unenrolled admin can reach nothing else.
  • Recovery codes — eight, shown once, stored hashed, single-use.
  • Sessions — short-lived, httpOnly + SameSite=Strict cookie, absolute 12-hour expiry, re-authentication required for super-admin-only actions.
  • Lockout — five failed attempts in fifteen minutes locks the account and emails every super admin. All attempts, successful or not, hit the audit log.

New tables

  • admin_users — id, email (unique), password_hash, totp_secret, totp_enrolled_at, role, is_active, created_by, last_sign_in_at, timestamps.
  • admin_recovery_codes — admin_id, code_hash, used_at.
  • admin_sessions — admin_id, token_hash, expires_at, ip, user_agent.
  • admin_audit_log — admin_id (nullable, for failed sign-ins), action, target_type, target_id, summary, ip, created_at. Append-only.
  • content_reports — [WIRED] reporter_id, target_type, target_id, reason, status.

All of them get RLS enabled with zero policies, like every other table — only the API touches them.

What the console must never do

  • Show a reader's private shelf, notes or unpublished reviews. It reads what another reader could see, plus aggregate counts. Support does not require voyeurism.
  • Hard-delete anything except an admin's own session. Hiding and merging are soft, reversible and logged.
  • Let an admin act on their own account in a way that could lock everyone out — no self-revoke, no self-demote, and the last super admin cannot be removed.

Open questions for the technology round

These mockups are deliberately stack-agnostic — every screen here is a table, a form, a card and a couple of CSS bars, which almost anything can render. The decisions worth making deliberately:

1. Where does it run? A separate small app on the existing Railway service (one more container, no new bill) versus a static SPA on Cloudflare Pages talking to the API. The second keeps hosting free but puts auth entirely in the browser; the first allows server-rendered, cookie-session admin pages with no token in JavaScript — which is the safer shape for a back office.
2. Rendered where? Server-rendered HTML from FastAPI (Jinja + a little htmx) reuses the API's own auth, models and Python, adds no build step and no new language — very much in the spirit of "cheap to run, cheap to maintain". A React/Vite SPA is more familiar and more interactive, at the cost of a second toolchain and a token in the browser.
3. Charts. Everything mocked here is achievable with CSS and inline SVG. A chart library is the first dependency that will want to creep in — worth resisting until a screen genuinely needs it.

Recommendation, for the next round: FastAPI + Jinja + htmx, served under admin.kitabi.in by the same Railway service. It reuses the session/auth code we would have to write anyway, keeps the admin token out of JavaScript entirely, adds no monthly cost and no second toolchain, and every screen above is comfortably within what server-rendered HTML does well.
